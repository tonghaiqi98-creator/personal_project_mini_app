# 轻量堂食扫码点餐小程序需求文档

## 1. 项目概述

### 1.1 项目名称

轻量堂食扫码点餐小程序 Demo

### 1.2 项目背景

本项目面向小餐馆、咖啡店、奶茶店、轻食店、档口等小型线下餐饮门店，提供一套轻量级堂食扫码点餐方案，做成一个一个模板，以后可进行小的修改就给不同的商家开发。

用户到店后扫描桌面二维码进入小程序，完成浏览菜单、选择商品、提交订单、模拟支付等流程。商家可在小程序内查看订单、接单、完成订单，并查看简单营业数据。

本项目第一版定位为 Demo 和可复用模板，主要用于：

1. 个人接单展示。
2. 后续客户交付模板。
3. 验证轻量扫码点餐业务闭环。
4. 为后续接入真实支付、云数据库、商家后台打基础。

### 1.3 项目目标

第一版目标完成一个可演示、可运行、可扩展的 MVP。

核心目标：

1. 顾客端能完成堂食扫码点餐流程。
2. 商家端能完成接单和订单状态处理。
3. 页面风格具备真实餐饮小程序质感。
4. 功能边界清晰，便于后续复制给不同门店。
5. 后续可扩展真实微信支付、云数据库、多门店、会员等能力。

---

## 2. 目标用户

### 2.1 顾客

顾客是到店堂食消费的用户。

顾客诉求：

1. 扫码后快速进入点餐页。
2. 清楚看到当前桌号。
3. 快速浏览菜单分类。
4. 方便选择菜品并加入购物车。
5. 清楚看到已选商品和总价。
6. 提交订单后能查看订单状态。
7. 能看到历史订单情况

### 2.2 商家 / 店员

商家或店员负责处理顾客提交的订单。

商家诉求：

1. 快速看到新订单。
2. 清楚知道桌号、商品、数量、备注。
3. 能对订单进行接单和完成操作。
4. 能看到今日订单数、营业额、热销商品等基础数据。
5. 能简单管理菜品上下架和价格。

### 2.3 项目开发者

项目开发者需要将该项目作为可复用模板，用于后续客户交付。

开发者诉求：

1. 项目结构清晰。
2. 功能范围可控。
3. 页面和数据模型便于扩展。
4. 后续可以低成本改成不同门店版本。
5. 代码和文档可被 AI 工具持续理解和维护。

## 3. 产品定位

本项目定位为：

> 面向小型餐饮门店的轻量堂食扫码点餐小程序模板。

第一版强调：

1. 轻量。
2. 易演示。
3. 易复制。
4. 易改造。
5. 先跑通核心流程。

不做大型餐饮管理系统，不与成熟餐饮 SaaS 做完整竞争。

## 5. 核心业务流程

### 5.1 顾客点餐流程

1. 顾客扫描桌面二维码。
2. 小程序打开门店首页。
3. 系统识别并展示当前桌号。
4. 顾客点击“开始点餐”。
5. 进入菜单页。
6. 顾客切换分类浏览菜品。
6. 点击菜品可看到菜品详情和口味/冷热/等情况
7. 顾客点击“+”加入购物车。
8. 底部购物车栏实时展示商品数量和总金额。
9. 顾客可以修改商品的数量和口味
9. 顾客点击“去结算”。
10. 进入确认订单页。
11. 顾客确认桌号、商品、数量、总价。
12. 顾客填写备注。
13. 顾客点击“模拟支付”。
14. 系统生成订单。
15. 订单状态变为“已支付，等待商家接单”。
16. 顾客进入订单详情页查看状态。

### 5.2 商家接单流程

1. 商家进入商家端首页。
2. 商家看到今日订单和待处理订单。
3. 商家进入订单管理页。
4. 商家查看新订单。
5. 商家点击“接单”。
6. 订单状态变为“制作中”。
7. 商家完成出餐后点击“完成订单”。
8. 订单状态变为“已完成”。
9. 今日营业数据同步更新。

### 5.3 菜品管理流程

1. 商家进入菜品管理页。
2. 商家查看当前菜品列表。
3. 商家可将菜品上架或下架。
4. 下架菜品不再展示给顾客。
5. 商家可修改菜品价格。
6. 修改后的价格在顾客端菜单页同步展示。

`pages/index/index`

让顾客确认当前门店和桌号，并进入点餐流程。

页面内容：

   * 文案：欢迎扫码点餐，下单后请等待店员送餐。

3. 今日推荐。

   * 展示 3 个推荐商品。
   * 每个商品包含名称、描述、标签、价格。

4. 底部主按钮。

   * 文案：开始点餐。
   * 点击后进入菜单页。

1. 点击“开始点餐”进入 `pages/menu/menu`。

| logo          | string | 门店 Logo |
| address       | string | 门店地址    |
| phone         | string | 联系电话    |
| businessHours | string | 营业时间    |
| status        | string | 营业状态    |

| updatedAt     | number | 更新时间    |

## 8. 订单状态设计

### 8.1 第一版订单状态

| 状态值       | 展示文案 | 说明               |
| --------- | ---- | ---------------- |
| paid      | 待接单  | 顾客已完成模拟支付，等待商家接单 |

### 8.2 状态流转

第一版状态流转：

`paid → accepted → completed`

异常状态：

`paid → cancelled`

第一版不做退款状态。

## 10. 非功能需求

### 10.1 性能要求

1. 首页打开无明显白屏。
2. 菜单页切换分类流畅。
3. 点击加购后金额和数量即时更新。
4. 页面交互不应出现明显卡顿。
5. 第一版数据量较小，暂不做复杂分页。

### 10.2 兼容要求

1. 适配常见 iPhone 机型。
2. 适配常见 Android 机型。
3. 底部按钮和购物车栏需要考虑安全区域。
4. 页面内容不应被刘海屏和底部手势条遮挡。

### 10.3 视觉要求

1. 页面风格统一。
2. 使用浅灰背景和白色卡片。
3. 保持足够留白。
4. 价格信息突出。
5. 主按钮使用统一品牌色。
6. 顾客端强调真实餐饮消费场景。
7. 商家端强调效率和信息清晰。

### 10.4 安全要求

第一版为 Demo，暂不做完整安全设计。

后续正式上线前需要补充：

1. 商家身份校验。
2. 用户 openid 获取。
3. 云数据库权限规则。
4. 订单创建权限。
5. 商家操作权限。
6. 支付回调校验。
7. 防止前端篡改订单金额。

## 16. 需求边界说明

本项目第一版必须坚持轻量化原则。

判断一个需求是否应该进入第一版，可按以下标准：

### 应该进入第一版

1. 是否服务于堂食扫码点餐主流程。
2. 是否能帮助 Demo 更容易被客户理解。
3. 是否能提升顾客下单闭环完整度。
4. 是否能提升商家接单闭环完整度。
5. 是否开发成本较低且不引入复杂依赖。
6. 是否便于简单修改后复用

### 不应该进入第一版

1. 是否需要真实资质或支付审核。
2. 是否明显增加后端复杂度。
3. 是否属于大型餐饮 SaaS 功能。
4. 是否会拖慢 MVP 交付。
5. 是否会带来大量售后维护成本。

第一版优先完成“能看、能点、能下单、能接单、能演示”。
