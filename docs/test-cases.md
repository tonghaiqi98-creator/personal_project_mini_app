# 验收用例

## 顾客扫码进店

前置条件：已生成包含 store_id 和 table_id 的桌号二维码。

步骤：

1. 顾客扫描桌号二维码。
2. 小程序打开。
3. 系统读取二维码参数。

预期结果：

- 进入对应门店菜单页。
- 页面显示正确桌号。
- 未识别到桌号时给出明确提示。

## 菜单浏览

前置条件：门店存在分类和上架菜品。

步骤：

1. 进入菜单页。
2. 点击不同菜品分类。
3. 查看菜品列表。

预期结果：

- 分类切换正常。
- 菜品名称、图片、价格展示正确。
- 下架菜品不展示。

## 菜品加购

前置条件：顾客位于菜单页。

步骤：

1. 选择一个菜品加入购物车。
2. 调整菜品数量。
3. 移除菜品。

预期结果：

- 购物车数量正确变化。
- 订单金额正确计算。
- 数量为 0 时菜品从购物车移除。

## 提交订单

前置条件：购物车内至少有一个菜品。

步骤：

1. 进入订单确认页。
2. 填写备注。
3. 点击提交订单。

预期结果：

- 创建订单成功。
- 订单金额与购物车一致。
- 订单状态为 pending_payment。
- 订单明细保存正确。

## 模拟支付

前置条件：订单已创建，状态为 pending_payment。

步骤：

1. 进入模拟支付页。
2. 点击模拟支付成功。

预期结果：

- 订单状态更新为 paid。
- 记录 paid_at。
- 顾客进入订单状态页。
- 商家端待接单列表出现该订单。

## 商家接单

前置条件：存在 paid 状态订单。

步骤：

1. 商家进入订单列表。
2. 打开订单详情。
3. 点击接单。

预期结果：

- 订单状态更新为 accepted。
- 记录 accepted_at。
- 订单从待接单列表进入制作中列表。

## 完成订单

前置条件：存在 accepted 状态订单。

步骤：

1. 商家进入订单详情。
2. 点击完成订单。

预期结果：

- 订单状态更新为 completed。
- 记录 completed_at。
- 订单进入已完成列表。

## 统计更新

前置条件：今日已有订单。

步骤：

1. 商家完成一笔订单。
2. 打开统计页。

预期结果：

- 今日订单数正确。
- 今日已完成订单数正确。
- 今日销售额正确。
- 待接单数量正确。

## MVP 验收边界

- 不验收配送流程。
- 不验收会员流程。
- 不验收库存扣减。
- 不验收打印机出单。
- 不验收多门店切换。
- 不验收真实微信支付。
