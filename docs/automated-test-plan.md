# 自动化测试计划

## 1. 当前阶段目标

当前项目仍处于本地 Mock MVP 阶段，暂未接入 CloudBase、真实微信支付和扫码桌号识别。

本阶段自动化测试目标是先保证以下内容稳定：

1. 购物车本地计算逻辑正确。
2. 订单本地状态流转正确。
3. 今日统计基于本地订单数据计算正确。
4. Mock 菜品数据结构可用。
5. `app.json` 路由与页面文件一致。

## 2. 当前阶段能自动化测试什么

1. 本地购物车加购、重复加购、数量增减、移除、清空、总数量和总金额。
2. 本地订单创建后的金额、桌号、备注、支付状态和状态流转。
3. 本地统计中的今日订单数、今日营业额、待处理订单数、已完成订单数、热销商品 Top 5。
4. Mock 菜品和分类数据的必填字段、唯一性、状态合法性、分类匹配关系。
5. 页面路由是否注册、页面文件是否完整、是否仍存在明显旧路径。

## 3. 当前阶段不能自动化测试什么

1. 微信开发者工具模拟器中的真实页面渲染效果。
2. 真实扫码识别桌号。
3. CloudBase 云数据库读写。
4. 云函数权限和并发处理。
5. 真实微信支付、退款和支付回调。
6. 真实用户 openid、登录态和商家权限。
7. 不同手机机型上的视觉适配和安全区表现。

这些内容需要在后续接入 miniprogram-automator、CloudBase 测试环境或微信开发者工具后补充。

## 4. 逻辑测试范围

逻辑测试目录：`tests/logic/`

当前脚本：

1. `cart.test.js`：覆盖购物车核心计算和清空逻辑。
2. `order.test.js`：覆盖订单生成、金额、桌号、备注、模拟支付状态和状态流转。
3. `stats.test.js`：覆盖今日统计和热销商品计算。
4. `mock-data.test.js`：覆盖本地菜品和分类 Mock 数据质量。
5. `routes.test.js`：覆盖 `app.json` 路由和页面文件完整性。

逻辑测试通过 Node 直接执行，不依赖微信开发者工具，不安装第三方测试框架。

## 5. E2E 测试范围

E2E 目录：`tests/e2e/`

当前只预留目录，不编写 miniprogram-automator 用例。

后续可补充以下 E2E 流程：

1. 顾客进入首页，点击开始点餐。
2. 进入菜单页，切换分类并加购商品。
3. 进入购物车页，调整数量和规格。
4. 进入确认订单页，填写备注并模拟支付。
5. 进入订单详情页，查看订单状态。
6. 商家进入订单页，接单并完成订单。
7. 商家统计页数据随订单变化更新。

## 6. 后续接 CloudBase 后的压测补充

接入 CloudBase 后，需要新增测试环境和压测脚本，重点覆盖：

1. `createOrder` 并发创建订单。
2. `updateOrderStatus` 并发更新同一订单时的幂等和状态保护。
3. `getMerchantOrders` 在订单量增加后的查询耗时。
4. `getStats` 在大量订单下的统计耗时。
5. 云数据库索引是否满足按门店、状态、日期查询。
6. 权限规则是否能防止顾客篡改订单金额或商家越权访问。

压测不在当前本地 Mock 阶段执行。

## 7. 测试执行顺序

建议执行顺序：

1. `mock-data.test.js`：先确认本地数据质量。
2. `routes.test.js`：确认页面路径和文件完整。
3. `cart.test.js`：确认购物车基础逻辑。
4. `order.test.js`：确认订单生成和状态流转。
5. `stats.test.js`：确认订单产生后的统计逻辑。

实际统一入口会按以下顺序执行：

1. `cart.test.js`
2. `order.test.js`
3. `stats.test.js`
4. `mock-data.test.js`
5. `routes.test.js`

运行命令：

```bash
node tests/run-tests.js
```

运行后会生成：

```text
tests/reports/local-test-report.md
```
